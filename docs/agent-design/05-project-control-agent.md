# 项目管控智能体（Project Control Agent，任务编号 0.x）

> 职责定义（产品负责人 2026-06 确认）：**推进项目进程，控制项目节点；协调多 Agent 任务拆解与上下文同步；质量管控触发人工介入，记录项目发展。**
> 落地依据：StatusCheckList「Schedule」sheet（W0–W10 周计划 + 优先级 + Owner）、Data Process 2.22「TaskLog/Timeline/数据处理流程」sheets（真实项目的任务管理实践）、各智能体 Gate 规则。

## 1. 定位

项目管控智能体是**平台的编排层（Orchestrator）**，不产出业务 Artifact，而是让其他四个智能体的工作"按正确的顺序、在正确的节点、由正确的人确认"地发生。它是 reference 中所有横切实践（周计划、TaskLog、优先级、两人互检、必问问题、预警升级）的产品化承载。

四大职责与产品能力映射：

| 职责 | 产品能力 |
|---|---|
| ① 推进项目进程，控制项目节点 | 阶段状态机 + 里程碑管理 + 排期/优先级调度 |
| ② 协调多 Agent 任务拆解与上下文同步 | 任务编排（DAG）+ 上下文总线 + Artifact 血缘传递 |
| ③ 质量管控触发人工介入 | Gate 路由 + 预警升级 + 人工介入收件箱 |
| ④ 记录项目发展 | 全程审计日志 + 修改留痕 + 知识回流 |

## 2. 职责 ①：项目进程与节点控制

### 2.1 阶段状态机

主流程五阶段（与前端 StageId 对应），每阶段由若干编号任务构成，阶段间有准入条件：

| 阶段 | 任务范围 | 准入条件（上一阶段出口） | 关键节点（里程碑） |
|---|---|---|---|
| S1 业务理解 | 1.0–1.34 | 项目立项 / SOW 签署 | M1：Scope+因子树+KBQ 确认（"6/3 业务给 input/output"类节点） |
| S2 数据治理 | 2.1–2.24 | M1 通过、数据需求已发出 | M2：数据验收完成（2.12 全量打分） |
| S3 数据校验与假设 | 2.31–2.35 | M2 通过 | M3：客户 sign-off + Model Input 就绪 |
| S4 建模 | 3.1–3.4 | M3 通过、先验确认 | M4：技术+业务双检验通过（"5/24 模型结果生成待校验"类节点） |
| S5 报告交付 | 4.1 | M4 通过 | M5：客户 Review 通过 |

回流路径是一等公民（不是异常）：纪要回流 S1 内循环；2.33 因子无可用指标 → 回 S1 找替代；2.34 反常识 → 回 2.31；3.3/3.4 不过 → 回 3.1/3.2；客户 Review 意见 → 回 S4/S5。状态机必须显式支持这些回边并记录回流原因。

### 2.2 排期与优先级调度

来自真实实践（Schedule + Timeline sheets）的调度要素：

- **周计划**：任务 × W0–W10 的甘特视图，含 Owner；
- **优先级分层**：数据 Task 分 1–4 级，按优先级确定检查顺序（数据检查 step1）；
- **每日指派**：Timeline sheet 实践为 任务 × 日期 × 执行人（XJ/TL/Beth）矩阵，含阶段标签（逻辑梳理/定义澄清/技术逻辑检查/开发/二次开发/Charting/客户 cross-check/内部穿插会&质量评估/客户 Review）；
- **延迟跟踪**：实例中记录了计划日 vs 实际偏移（-4、-2、0），调度器需计算并预警延迟。

### 2.3 节点控制规则

- 里程碑只能由人（项目负责人/客户）关闭，AI 可建议"就绪"；
- 未达准入条件的阶段任务不可启动（如未 sign-off 的数据不得入模）；
- 关键路径上的任务延迟 → 自动重排下游 + 通知 Owner。

## 3. 职责 ②：多 Agent 任务拆解与上下文同步

### 3.1 任务拆解

- 接收阶段目标 → 按编号体系实例化任务（如收到数据需求回执后，把"数据处理"拆解为 Task1–Task37 粒度的 TaskLog，每个 Task 绑定：数据来源、对接人、服务的 L4 因子、优先级、Owner）；
- 拆解模板来自知识库（StatusCheckList 即任务模板库），项目启动时按行业包+Scope 实例化；
- 支持动态增补（如访谈后新增因子 → 自动派生新的数据 Task 与校验任务）。

### 3.2 上下文同步（Context Bus）

多 Agent 协作的核心问题是上下文一致性。同步内容与机制：

| 同步对象 | 生产者 | 消费者 | 同步时机 |
|---|---|---|---|
| 因子树版本 | 商业智能体 | 全部 | 每次 Gate 确认后广播新版本号；下游任务自动标记"基于过期版本"待刷新 |
| Scope 约束 | 商业智能体 | 全部 | 项目级常量，变更需重走 G1.0 |
| 数据口径备注 / 客户 Q&A 结论 | 数据智能体 | 模型/报告 | 沟通表回执即同步（如"WD 按 NAB 销售额加权"成为全局口径注释） |
| 先验登记表 | 模型智能体 | 报告（解读）、商业（业务检验） | G3.1 后冻结，变更留痕 |
| 模型结果版本 | 模型智能体 | 报告 | 每轮训练产生带版本号的结果集，报告只消费"双检验通过"版本 |
| KBQ 状态 | 商业智能体 | 报告 | KBQ→证据图表的覆盖率追踪 |

实现要求：每个 Artifact 带 `version + producedByTask + 基于的上游版本`（血缘三元组），管控智能体在任务启动前做**版本一致性检查**，不一致即阻塞并提示刷新。

## 4. 职责 ③：质量管控与人工介入触发

管控智能体不制定质量规则（规则在各智能体的规则库），但**统一执行触发与路由**：

### 4.1 触发规则汇总（全平台 Gate 路由表）

| 来源 | 触发条件 | 路由对象 | 紧急度 |
|---|---|---|---|
| G1.2 | 因子增删超 10–20% 阈值 | Solution Team 联审 | 中 |
| G2.12 | 指标打分 = 0.5 | 数据 Owner 决策 | 中 |
| G2.12 | 某 L4 因子全部指标 = 0 | BA + 客户（找替代数据） | **高（预警）** |
| G2.22 | 处理逻辑待确认 / cross-check 不一致 | 数据 Owner（两人互检） | 中 |
| G2.22 | YoY GR% >30% / 月度异常值 | 数据 Owner → 客户沟通表 | 中 |
| G2.32 | 客户 sign-off 拒绝 | 数据 Task 回炉 | 高 |
| G2.33 | 指标总分 1.5–3（Acceptable） | DS 确认是否入模 | 低 |
| G3.3 | Negative Base / Negative Coeff | DS，**红旗即停** | **高** |
| G3.3 | 五指标任一超出 benchmark | DS 诊断 | 中 |
| G3.4 | 弹性偏离先验 >30%（红灯） | BA + DS 会商（调优 or 重审先验） | 高 |
| G4.2 | 客户 Review 意见 | 按意见类型路由回 S4/S5 | 高 |

### 4.2 人工介入收件箱（Gate Inbox）

- 每条触发生成一个待办：证据 Artifact + AI 建议 + 候选处置选项 + 驳回回炉目标任务；
- SLA 跟踪：超时未处理升级到项目负责人；
- 决策记录：选择 + 理由 + 决策人 + 时间，写入审计日志。

## 5. 职责 ④：记录项目发展

### 5.1 审计日志（Event Log）

记录全部事件：任务状态变迁、Agent 调用（含 LLM 模型版本——产品要求 LLM 可切换，必须记录每次产出用的是哪个模型）、Gate 决策、回流原因、Artifact 版本变更、客户确认记录。

### 5.2 修改留痕（与知识回流联动）

StatusCheckList 原文要求："需要有个地方记录每个项目都因为什么补了什么，或者删了什么，后续需要以此来维护知识库。"

- 因子/指标增删改：项目 + 内容 + 原因 + 来源（AI推荐/报告/访谈）+ 确认人 + 时间；
- 项目结束时管控智能体发起**知识回流评审**：哪些项目定制应升级进 Solution 沉淀区（任务 1.4/1.5 的触发者）；
- 数据处理逻辑的多轮迭代 LOG（参照 TaskLog 的 1225/1230/0104 多版本 PROCESSING LOGIC 实践）。

### 5.3 项目记忆（跨会话上下文）

为长周期项目（W0–W10）维护项目级摘要：当前阶段、未决 Gate、关键口径结论、客户偏好——作为各 Agent 会话的注入上下文，避免重复询问已确认事项。

## 6. 能力（工具）定义

| 能力 | 说明 | 类型 |
|---|---|---|
| `project_init` | 按行业包 + SOW 实例化任务计划（编号体系 + 周计划 + Owner） | 模板引擎 |
| `task_scheduler` | 优先级调度、依赖检查、延迟重排与预警 | 调度器 |
| `context_sync` | Artifact 版本广播、血缘一致性检查、过期标记 | 事件总线 |
| `gate_router` | 按路由表派发人工介入待办，SLA 跟踪与升级 | 工作流 |
| `audit_logger` | 全事件记录（含 LLM 模型版本） | 日志服务 |
| `change_tracker` | 因子/指标/口径/先验的修改留痕 | 主数据服务 |
| `knowledge_review` | 项目收尾的知识回流评审发起 | 工作流 + LLM |
| `status_reporting` | 项目状态摘要生成（对内站会 / 对客户进度） | LLM + 模板 |

## 7. Gate 定义（管控智能体自身的人工节点）

| Gate | 触发点 | 判定者 |
|---|---|---|
| G0.1 项目计划确认 | 任务计划实例化完成 | 项目负责人 |
| G0.2 里程碑关闭 | M1–M5 各节点 AI 判定"就绪" | 项目负责人（M3/M5 需客户） |
| G0.3 知识回流评审 | 项目收尾 | Solution Team |

## 8. 与其他智能体接口

管控智能体与所有智能体双向连接：下行（任务派发、上下文注入、版本校验）/ 上行（任务状态、Gate 触发、Artifact 注册）。各业务智能体不直接互调，**一切跨 Agent 传递经过管控智能体的上下文总线**——这保证血缘完整与审计可追溯，也是 LangGraph 编排图的中心节点设计依据。

```
                 ┌─────────────────────────────┐
                 │   项目管控智能体（编排层）      │
                 │ 状态机 · 调度 · 上下文总线      │
                 │ Gate路由 · 审计 · 知识回流     │
                 └──┬──────┬──────┬──────┬─────┘
                    ↕      ↕      ↕      ↕
                 商业Agent 数据Agent 模型Agent 报告Agent
                    ↕      ↕      ↕      ↕
                 ┌─────────────────────────────┐
                 │  共享层：因子树主数据 · 知识库   │
                 │  Artifact仓库（含血缘） · 规则库 │
                 └─────────────────────────────┘
```
