# 数据智能体（Data Agent，任务编号 2.x）

> 源材料：`reference/02.数据智能体/` 全量（Data Validation_2.1、Data Process_2.21–2.24、因子树+下钻维度+交叉检验_2.31、Data Analysis_2.32、Data statistical test_2.33）。

## 1. 定位

数据智能体是平台**规则最完备**的智能体，承担从"客户交来的原始数据"到"可建模 Model Input"的全部治理工作，形成**两道质量关卡 + 一条集成管线 + 一道业务/技术双重交叉校验**：

```
原始数据 ─→ [关卡1] 2.1 通用校验打分 ─→ 2.2 集成管线（宽表/字典/数据集）
        ─→ [双重校验] 2.31 业务校验六步法 + 2.32 Charting客户sign-off + 2.33 统计校验
        ─→ [关卡2] 2.34 OLS 预验证 ─→ 2.35 Model Input
```

核心信条（贯穿 2.11）："Rubbish in, rubbish out" —— 数据质量即模型质量。

## 2. 任务 2.1：Data Validation（通用校验与打分）

### 2.11 四大校验规则

| 规则 | 核心 | 细分项 |
|---|---|---|
| **Consistency 一致性** | 无矛盾、统一标准 | ①维度一致性（统计维度统一，如投放金额均按实际到账；渠道/产品/时间划分标准统一）②时间一致性（粒度统一、范围对齐）③口径一致性（计算逻辑与数据源统一且不随意变更） |
| **Accuracy 准确性** | 真实、无误差 | ①数值准确性（无录入/计算错误，无负数/不合理波动；可被第二来源交叉验证，如财务 vs 营销口径）②业务准确性（与财务或其他系统统计口径一致） |
| **Comprehensiveness 完整性** | 无缺失、无遗漏 | ①字段完整性（核心字段齐全）②数据完整性（建模颗粒度/时间颗粒度上无缺失） |
| **Granularity 颗粒度** | 适配建模、可拆解 | ①时间颗粒度（越细越好，建模最低月度）②模型颗粒度（适配 L4 的建模颗粒度）③下钻颗粒度（L5–L8 下钻维度） |

### 2.11 打分细则（0 / 0.5 / 1 三态）

| 评分项 | 0 分 | 0.5 分 | 1 分 |
|---|---|---|---|
| 维度一致性 | 不一致比例 >10% | ≤10% 且对模型影响小 | 完全统一、定义清晰 |
| 时间一致性 | 粒度混乱或月度以下 | 少量不一致 ≤10%、月度以上 | 统一粒度、月度之上 |
| 口径一致性 | 口径/源变化且前后半年趋势同比变化 >30% | 变化 ≤30% | 全程统一或变更有严格记录 |
| 数值准确性 | 明显错误占比 >10% | 错误占比 5%–10% | 完全正确或误差 <5% |
| 业务准确性 | 与财务/系统口径偏差 >10% | 偏差 5%–10% | 偏差 <5% |
| 字段完整性 | metrics 无范围内字段 | 只有 spending 无 performance | 字段齐全 |
| 数据完整性 | 缺失 ≥10% | 缺失 <10% | 无缺失 |
| 时间颗粒度 | 季度/年度 | — | 月/周/日 |
| 模型颗粒度 | 无法适配 L4 建模需求 | 基本适配但需聚合/拆解 | 完全适配可直接建模 |
| 下钻颗粒度 | 无法下钻或维度与业务无关 | 仅 2 个下钻维度 | >2 个下钻维度（L5–L8） |

### 2.12 数据质量评分（验收判定）

按 **L1×L2×L3×L4×指标** 逐行打分（一致性/完整性/颗粒度/Brand 颗粒度/Channel 颗粒度/真实性 + Total Score），验收规则：

| Final Score | 处置 |
|---|---|
| 1 | 验收通过 |
| 0.5 | **人进入决策如何处理**（人工介入 Gate） |
| 0 | 数据完全不能用，该指标直接弃用并**预警**；若某 L4 因子仅有的指标都不通过 → 升级预警 |

补充验收标准（评分 sheet 头部）：Consistency（口径未变=1）、Frequency（周/月=1，年=0）、Data coverage（>2年=1，1–2年=0.5，<1年=0）、Brand/Channel/Geography integrity（完整=1，不完整=0.5，缺失=0）。

## 3. 任务 2.2：Data Process（集成管线）

### 2.21 宽表数据集维度（统一长表 Schema）

所有数据 Task 输出统一的长表结构（这是平台数据层的核心契约）：

| 列 | 定义 | 规则 |
|---|---|---|
| TASKNAME / 数据源方 / 数据源 / 数据分类 | 任务与来源标识 | 定位数据源 |
| 品牌 / 产品线 / 产品 | 模型颗粒度 | Product 主数据表清洗统一命名，不满足填 NA |
| 大区 / 省份组别 / 省份 | 模型颗粒度 | Geo 主数据表清洗 |
| 渠道类型 / 渠道 | 模型颗粒度 | Channel 主数据表清洗 |
| 年 / 月 | 时间颗粒度 | Time 主数据表清洗 |
| 数据类型 Level1–Level4 | 对应因子树 L1–L4 | 按因子结构树命名 |
| 数据类型 Level5–Level8 | 因子下钻颗粒度 | L4 的下钻维度，没有填 NA |
| METRICS类型 / METRICS / Unit / VALUE | 指标与值 | Unit：percentage / Unit / Volume K / RMB |

→ **因子树即数据 Schema**：L1–L8 直接成为长表的列，业务结构与数据结构同构。

### 2.22 数据处理（TaskLog 工作流）

Danone 案例拆出 **37 个数据 Task**（Task0 Market segmentation ~ Task37 PPI），每个 Task 在 TaskLog 中管理：

`优先级 | Task No. | TASKNAME | 数据来源（Manual-德勤/OSS）| 数据逻辑对接人 | 数据问题 | 数据反馈 | 数据处理 | LOG（按日期多轮）| PROCESSING LOGIC（多轮迭代 1225/1230/0104）| Source_Table | Task_Summary | 处理进度 | Owner | 备注`

**人机协作核心**："人交互确认数据处理逻辑" —— 处理逻辑（如渠道 mapping、产品线匹配、大区分类、负数体现）由 AI/工程师提出，按日期多轮迭代记录，人工 check 后标记"已check / 完成开发"。

数据处理七步流程（数据处理流程 sheet）：

1. 初步梳理数据逻辑；
2. 定义澄清 & 补充数据；
3. 数据补充后重新梳理逻辑；
4. by task 导出数据后检查（a. 逻辑/字段/名称匹配；b. 趋势是否缺失；c. 判断源数据问题 vs mapping 问题）；
5. 逻辑更新后重新导出，做 charting；
6. 与源数据 cross-check（**需两人互检**）；
7. 寻找异常点原因。

数据检查必问三题（Timeline sheet）：①YoY GR% 超过 30% 需要知道原因；②YoY 花费趋势 vs KPI 趋势及 unit cost 趋势；③by month 异常值。

### 2.23 数据字典（ODS→DW 元数据）

每张源表登记：Theme（ODS/DW）/ SheetName / Source Sheet 与位置（OSS 路径或 database）/ 来源系统（魔方/手工数据）/ granularity / **服务于哪一个 L4 factor** / ETL 逻辑 / 时间范围 / **Y or X** / TableName / Data source DEPT / Data provider。

每张 DW 表的列级 ETL 逻辑四类：`Hardcode`（常量）/ `Mapping`（主数据表映射）/ `Transform`（如日期格式、渠道 case when）/ `Calculation`（如 PPI = 促销零售价合计 ÷ 原零售价合计）。

配套主数据 mapping 表：销售渠道、EC 渠道、产品组拆分、产品线、UPC、Social 产品品类、地域、Media 品类、省份城市。

## 4. 任务 2.3：数据交叉校验

### 2.31 业务校验（六步法）

> 定位（原文）：在数据清洗后、模型跑数前，用业务逻辑对数据做一次"体检"和"校准"，把业务常识转化为模型约束，确保模型结果"不反常识、可解释、可落地"。

| 步骤 | 内容 | 输出示例 |
|---|---|---|
| 1. 全景概览 | 整体销量/销售额趋势（3年月度）、同比增速 vs 大盘、份额变化、费用×销量叠图、品牌搜索趋势 | "2025YTD 销量+3% 但份额 30.5%→29%，跑输品类（+5%）；搜索量 -15%" |
| 2. 维度拆解 | 渠道（占比/ROI/铺货/SPPD）、产品（Top SKU/新品/包装）、区域（份额×增长四象限：牛眼/成熟/潜力/问题市场）、**营销活动（核心重点**：by platform/campaign/format/influencer tier/event/promotion type） | "抖音 ROI 1.2，天猫 2.0；头部达人 CPE 8 元 vs 腰部 2 元" |
| 3. 异常定位 | 偏离阈值（ROI 降幅>30% 或销量波动>20%）自动沿 **L5–L8 下钻维度**下钻 | "2025年8月，抖音×头部达人×产品演示内容，ROI 2.1→1.2（-43%）" |
| 4. 假设生成 | 基于异常+行业知识库生成业务原因假设（带置信度+验证方法+涉及因子） | "①内容老化②粉丝重合③竞品大促（高置信）" |
| 5. 先验提取 | 客户确认后的假设 → 模型数学约束 | "抖音弹性 Normal(0.45,0.1) 截断至 [0.3,0.7]；竞品交互项 β~Normal(-0.15,0.05) 截断至负" |
| 6. 客户核对 | 带假设与约束问客户："您觉得对吗？" | 客户确认：8 月抖音代理更换，同意弹性上限 0.7 |

> 注：步骤 5（先验提取）在当前实施中标注"先不做"，但规则已定义，是 3.1 先验设置的直接上游。

下钻维度登记表（Sheet3/4）给出全部 93 个 L4 指标的下钻配置（by platform / by competitor / by channel / by campaign / by influencer tier / by promotion type / by region / by event / by format / by location…），是异常定位的导航图。

### 2.32 数据展示（Charting + 趋势解读 + 客户确认）

- **Charting 工作簿**：Mapping、Summary（按部门/年度花费）、PIVOT（总/Region/店类型/产品）、KPI/Base/PFME(花费)/Promotion 分区趋势、MODEL Design、全量长表；
- **趋势解读 Deck**（184 页实例）：Session 结构 = ①KBQ Alignment ②Data Review（Sales/Base/Spending/Promotion 分区，每页含数据 + 解读 + **Sign-off Y/N 栏** + 备注假设）③Model Design Review；
- **数据&指标沟通表**：`问题 | 负责人 | 反馈` 三列的客户 Q&A 工作流（如 WD 加权逻辑、RSP 是否含折扣、批发数据获取方式、SKU 定义等），是数据口径确认的一等公民 Artifact。

### 2.33 统计校验（指标技术校验）

对每个 **L1–L4×指标** 计算三项检验并打分：

| 检验 | 方法 | 0 分 | 0.5 分 | 1 分 | 2 分 |
|---|---|---|---|---|---|
| 波动性 | CV =方差/均值（数据先缩放至 0–1） | cv≤0.05 极小，建议剔除/换数据 | 0.05–0.1 弱，非业务不可替代不建议入模 | 0.1–0.2 中等 | ≥0.2 高 |
| 相关性 | Pearson r（因子 vs KPI 单变量） | \|r\|<0.1 基本不相关 | 0.1–0.3 弱相关（因子少或业务重要时入模） | 0.3–0.5 较相关，建议入模 | ≥0.5 强相关 |
| 共线性 | VIF | =1 无共线 | 1–5 中等可接受 | ≥5 明显需关注 | ≥10 严重必须处理 |

**Total = CV + Pearson + VIF**：≥3 Good；1.5–3 Acceptable；<1.5 unconsiderable（不入模）。

业务规则（StatusCheckList 原文）：一个因子有多个指标时，部分指标在此被筛掉；**若一个因子仅有的指标都不通过 → 预警**（回到商业智能体/客户寻找替代数据）。

### 2.34 指标筛选（OLS 快速验证）

用 OLS 快速验证：选择什么指标可以使各 L4 因子的 **Contribution 和 ROI Range** 符合业务预期（参照行业知识中的弹性典型范围与 2.31 形成的业务假设）。这是"模型前的模型"——以低成本验证业务层级 sense 与数据的一致性。

### 2.35 Model Input

根据选定指标，从 2.24 数据集生成最终建模输入（按模型对象拆分：渠道 × 区域组 × 产品组）。

## 6. Gate 定义

| Gate | 触发点 | 判定者 | 规则 |
|---|---|---|---|
| G2.12 数据验收 | 每批数据打分完成 | DS（0.5 分项）/自动（1、0 分项） | 1 自动过；0.5 人决策；0 弃用+预警；因子全指标 0 → 升级 |
| G2.22 处理逻辑确认 | 每个 Task 的 PROCESSING LOGIC 迭代 | 数据 Owner | 多轮 LOG 留痕，"已check"标记；两人互检 cross-check |
| G2.32 客户 Sign-off | 趋势解读 Deck Review | 客户 | 每页 Y/N；未 sign-off 的数据不得入模 |
| G2.33 指标入模 | 统计打分完成 | DS | <1.5 剔除；1.5–3 人确认；因子无可用指标 → 预警回流 |
| G2.34 指标选型确认 | OLS 预验证完成 | DS + BA | Contribution/ROI Range 反常识 → 回 2.31 重新假设 |

| 方向 | 内容 |
