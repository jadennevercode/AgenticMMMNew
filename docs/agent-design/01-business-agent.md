# 商业智能体（Business Agent，任务编号 1.x）

> 源材料：`reference/01.商业智能体/` 全量 + StatusCheckList「流程大框架」sheet + 产品截图反馈（2026-06-11）。

## 1. 定位

把**非结构化的商业输入**（SOW、行业知识、品牌财报/研报、分层访谈）转化为**结构化的可建模资产**：

- 项目范围与分析颗粒度（Scope）；
- 业务因子结构树 L1–L4 及对应指标库；
- 关键业务问题（KBQ）清单；
- 数据需求文档与数据采集模板。

商业智能体是整条流水线的**业务真值来源**：后续数据校验的"业务准确性"、模型的"业务检验（3.4）"、报告的"商业逻辑补充"全部回溯到这里产出的因子树与 KBQ。

## 2. 职责边界

| 做 | 不做 |
|---|---|
| 因子/指标的召回、补充建议与增删管理 | 数据本身的校验与处理（数据智能体） |
| KBQ 生成、访谈提纲生成、纪要消化 | 模型先验的数学形式化（模型智能体，但 2.31 的先验提取依赖本体输出的业务假设） |
| 数据需求模板生成 | 数据采集执行 |
| 行业/品牌知识沉淀与主数据管理（1.4/1.5） | 报表渲染 |

## 3. 工作流程（流程大框架五步）

### Step 1 — Scope（1.0）

**输入**：SOW、客户合同。
**规则**（来自 Scope.xlsx，为保证项目时效的硬约束）：

1. 选择 1 条产品线或 1 个产品线组；
2. 渠道数 ≤4：1 个线上渠道（线上含多平台：天猫/京东/拼多多/抖音/快手等）+ 3 个线下渠道（MT 现代商超 / TT 传统流通 / AFH 新兴渠道）；
3. 地区和平台 ≤6 组：地区数针对线下（各省市分为 ≤6 组），平台数针对线上。

**输出**：Scope 表（渠道 × 地区和平台 × 产品 矩阵）。
**人机协作**：AI 依据 SOW 起草 → BA 确认（Gate）。

### Step 2 — Factor（1.21）

**输入**：行业知识库（1.1）、Scope。
**召回规则**：

| 层级 | 规则 |
|---|---|
| L1/L2 | 所有行业一致，**100% 知识召回**，不可修改 |
| L3/L4 | 根据行业知识召回，**默认全部都要**；可补充 10%–20%，来源：A. AI 推荐；B. 品牌报告/各类报告；C. 后续访谈；**每个补充需人确认** |

**留痕要求（原文）**：需要有个地方记录每个项目因为什么补了什么、删了什么，后续以此维护知识库。
**输出**：项目因子树 v1（L1–L4），每条记录含来源标记（知识库召回 / AI 推荐 / 报告补充 / 访谈补充）与确认人。

行业知识 → L3/L4 的生成逻辑（行业知识.xlsx 的三段式）：

1. **行业底层逻辑** → 生成 L1 生意基本盘-L2 外部因素 的 L3/L4（宏观环境、品类趋势、竞争格局）；
2. **行业常用营销打法矩阵**（拉新/促活/提升客单/防御竞品/品牌建设 × 打法）→ 生成 L1 消费者需求驱动 / 渠道成交驱动 / 促销优惠 的 L3/L4；
3. **除营销外核心驱动因子拆解**（销量 = 铺货基数 × 卖力效率 × 复购粘性 × 营销效果）→ 生成 L1 生意基本盘-L2 内部因素 的 L3/L4（动销、复购粘性）。

行业知识还携带可作为先验参考的**弹性典型范围**（媒体弹性 0.1–0.5、促销弹性日常 0.5–1.2、品牌搜索弹性 0.2–0.4 滞后 1–2 周、内容互动弹性 0.1–0.3）和指标经验值（ND>80% 边际递减、SOS 每 +1% 销量 +0.5–1%、复购率每 +10% 销量 +8–12% 等），供 2.31 先验提取与 3.1 先验设置引用。

### Step 3 — Indicator（1.22）

**规则**：

- 知识库中已有的 L4：**默认所有指标都要，100% 召回**，可增删 10%–20%；
- 知识库中没有的 L4（新补充的）：**AI 补充对应指标**。

**输出**：因子-指标宽表。每个指标携带：指标选择、最细颗粒度（By time/brand/category/channel/region/format/campaign…）、相关业务团队、数据来源、五项数据准入标准（完整性≥2年理想3年 / 准确性可被业务确认 / 颗粒度 daily~monthly 不接受 yearly / 波动性需有变化 / 一致性口径可比）。

**产品反馈**：indicator 视图必须显示所属 factor 上下文，呈现"一张宽表串起所有信息"的感觉。

### Step 4 — Interview（1.3，KBQ 闭环）

KBQ 三步法（KBQs.xlsx Step1–3）：

1. **AI 读品牌报告**（如 CICC Project Aurora 竞品 Benchmark，密码 aurora）回答 6 类业务问题，总结为 Knowledge：
   - ①消费者需求因子：多少种营销投资带动需求影响生意？
   - ②渠道成交因子：渠道上做了哪些商业行为？
   - ③促销优惠因子：促销的具体方法和机制？
   - ④生意基本盘因子：内外部关注的行业大环境因子？
   - ⑤生意表现复盘：过去几年表现、增长机会与痛点（渠道/产品线/区域维度）？
   - ⑥未来预算优化：未来财年媒体预算与生意目标？
2. **AI 输出业务现状与业务问题假设**，生成访谈提纲找客户确认；
3. **AI 发现报告中涉及因子树 L3/L4 没有的内容 → 主动上报**（因子树补充建议）。

数据部分访谈方向（针对每个 L4）：①这些指标是否都有；②有的话在哪、多长时间、是否满足数据准入标准。

**分层访谈结构**（Interview Plan，仅参考对象层级结构）：

| 层级 | 对象 | 问什么 | 时长 |
|---|---|---|---|
| Layer 1 | Leadership（GM） | 痛点与业务战略、投资策略 3 年转变、KBQ 期望、分析范围 | 60min |
| Layer 2 | Management（Mkt/Sales/Finance） | 业务细节、KBQ 下钻、数据可行性 high-level | 60min |
| Layer 3 | Operation（Brand/Media/Activation/EC/KA/TradeMkt/Execution/SIA/O2O/RTM） | 数据明细：覆盖、颗粒度、指标定义、连续性、获取方式 | 90min |

提纲问题统一三段式标签：`[业务现状理解]` / `[关键业务问题与期望]` / `[数据颗粒度与范围]`。

**纪要回流**：访谈完成上传纪要 → Agent 比对纪要与因子树/指标库 → **AI 标出要改什么（增删 L3/L4、调指标、数据口径备注），人确认** → 刷新 Step 2/3 产出。

KBQ 最终形态（KBQs原 sheet）：编号 + 问题分类（Type 1 价值回报评估 / Type 2 投资效率分析 / Type 3 投资规划 + Themed Topics：场景投资效率、未来规划模拟、外部因素影响）+ 关键业务问题 + 访谈上下文 + 提出方。Danone 案例沉淀了 A1–A24 共 20+ 条真实 KBQ 可作种子数据。

**产品反馈**：KBQ 列表要声明"仅是 typical 的问题列表 for reference，可不 follow"；产品流程中没有独立的 KBQ 步骤，**在 Interview 环节内解决**。

### Step 5 — Data Request（1.23）

**规则**：确认完 Step 2/3 后，**以 L3 为表、L4 为 sheet、指标颗粒度 + 指标为列**生成数据需求文档。
factor&data_request 中沉淀了 12+ 个模板实例（Mkt Activation Buzz、Market dominance、salesforce、Omni-channel、竞品 RSP、Media、线条饮料、Price&cost、Competitor Sales+Promo、Store Execution、Macroeconomics、BHT），统一结构：

```
Data Request
Time Coverage / Granularity / Brand Coverage
表头：Date + 维度列（brand/category/channel/format/region…）+ 指标列
```

收数后移交数据智能体按 2.11 检查规则初步诊断。

### 辅助产出

- **Seasonality Calendar**：按月 × 业务条线（SIA/线下 Sales/EC/O2O/社区团购）的季节性事件日历（流感、春节、五一、开学季、双十一、供暖…），供模型季节性变量与 dummy 设置；
- **Competitors List**：竞品 × 数据可得性矩阵（价格/SKU/渠道/EC/媒体/Social/门店执行 × 数据源），含选取规则：分 4 类（FD/CSD/Juice/RTDT）、与本品产品线 mapping（只能 map 一个，多个则加总/平均）、模型不显著则剔除；
- **模型分类及应用场景（1.24）**：见 [03-model-agent.md](03-model-agent.md) §2。

## 4. 能力（工具）定义

| 能力 | 说明 | 类型 |
|---|---|---|
| `scope_drafting` | SOW → Scope 表草案（带约束校验：渠道≤4、组≤6） | LLM + 规则 |
| `factor_recall` | 按行业包召回 L1–L4 + 指标，输出带来源标记 | 知识库检索 |
| `factor_suggest` | 基于报告/纪要推荐 L3/L4/指标补充（≤20% 阈值控制） | LLM |
| `kbq_answering` | 读品牌报告回答 6 类问题，生成 Knowledge 摘要 | LLM（长文档） |
| `interview_guide_gen` | 按层级 + KBQ 生成访谈提纲 | LLM + 模板 |
| `minutes_diffing` | 纪要 vs 因子树比对，产出结构化改动建议（diff） | LLM |
| `data_request_gen` | 因子树 → 数据需求文档（L3 表/L4 sheet/指标列） | 模板引擎 |
| `knowledge_writeback` | 增删改记录回写知识库（1.4/1.5） | 主数据服务 |

## 5. Gate 定义

| Gate | 触发点 | 判定者 | 证据 | 驳回回炉 |
|---|---|---|---|---|
| G1.0 Scope 确认 | Scope 草案完成 | BA + 客户 | Scope 表、SOW 对照 | 重新起草 |
| G1.2 因子树确认 | 召回+补充完成 | BA（超 20% 增删 → Solution Team 联审） | 因子树 diff、来源标记 | factor_suggest |
| G1.3 纪要改动确认 | 每份纪要消化后 | BA | 结构化 diff 列表 | minutes_diffing |
| G1.23 数据需求发出 | 模板生成后 | BA + 客户对接人 | 数据需求文档 | data_request_gen |

## 6. 与其他智能体接口

| 方向 | 内容 |
|---|---|
| → 数据智能体 | 因子树 + 指标库 + 数据准入标准（2.11 校验的对象与标准来源）；数据需求模板 |
| → 模型智能体 | KBQ（模型设计目标）、业务假设与弹性典型范围（3.1 先验素材）、Seasonality Calendar、模型选型建议（1.24） |
| → 报告智能体 | KBQ（报告必须回答的问题）、业务上下文（解读话术的商业逻辑补充来源） |
| ← 数据智能体 | 2.31 业务校验发现的异常假设 → 客户核对问题清单 |
| ← 项目管控智能体 | 任务派发、Gate 升级、修改留痕归档 |

## 7. Danone 案例沉淀的领域知识（种子数据）

访谈纪要中可直接产品化的结构化知识：

- **渠道体系**：MT（含 CVS/大卖场/折扣店）、TT、AFH（特渠五大场景"金钻五部曲"：运动/学校/工矿/白领/景区出行）、批发（建议与 TT 合并：AFH 投资不作用于 AFH 批发，避免稀释有效性）、EC（分销/自营/旗舰店/抖音）、O2O（前置仓/闪电仓贡献 40% 销量）；
- **区域体系**：TOP10/TOP40/Others 城市分级（购买力/人口/品类增速/GDP 四维），县城 kmeans ABC，乡镇按人口 abc；control 店约 100 万家占 55% 销售，uncontrol 占 45%；
- **数据源 mapping**：Nielsen（MT+TT，含 control+uncontrol offtake）、Kantar（AFH）、Smartpath（EC&O2O）、朗镜（门店执行，仅 2025/04–09）、BHT（仅 A/B 城市，月度断档）、魔方/Compass（内部）；
- **关键业务判断**：媒体 vs 终端执行相对贡献是 KBQ 核心（"只比媒体渠道就没用"）；终端执行三维度 Availability/Visibility/Impact；uncontrolled store 是真正的学习场；TTS 提成占比 3 年从 10% 变到 80%，同额费用业务含义不同需建模处理；O2O 对线下有分流需考虑渠道迁移。

这些应作为饮料行业知识包 + Danone 项目定制区的初始内容。
