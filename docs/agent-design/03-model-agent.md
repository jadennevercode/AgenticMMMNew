## 1. 定位

把数据智能体产出的 Model Input 与商业智能体沉淀的业务假设，转化为**经过技术与业务双重检验的可信模型**。四个任务构成"设置 → 调优 → 双检验"闭环：

```
3.1 先验设置（业务假设→数学约束） ─→ 3.2 调优（调参寻优）
   ─→ 3.3 技术检验（统计指标过关？） ─[红旗即停]→ 3.4 业务检验（符合先验？）
   不过 → 回 3.1/3.2 迭代；通过 → 移交报告智能体
```

职责分工（StatusCheckList）：3.1/3.4 偏业务（Sylvia），3.2/3.3 偏技术（Yifei/DS）。

## 2. 模型选型（任务 1.24，商业智能体输入）

| | 概率模型（Meridian、PyMC） | 线性拟合（OLS） |
|---|---|---|
| 使用场景 | 数据满足业务层级 sense、需最终落地、每个层级因子都要高精度结果 | 需要快速迭代出 ROI / Contribution 结果 |
| 优点 | ①可输入业务先验引导模型结果；②精度高、可控性强 | ①对数据质量要求不高；②快速简单；③能输出各层级因子大概结果，快速验证业务层级 sense 与数据是否一致 |
| 缺点 | ①调参时间长；②训练成本高 | 准确性不够高 |
| 平台角色 | 正式建模（3.x 主路径） | 2.34 指标预验证 +快速迭代模式 |

**贝叶斯层级模型设计实例**（2.32「6.MODEL Design」，医学营养品案例）：

- 模型对象拆分：产品线（AAA / Nutrition）× 渠道（offline / online）；
- 层级（hierarchy）：offline 按 region，online 按 storetype，下层 Product channel；
- 变量设计：自有 PFME by Level3、product halo by Level2、store halo（EC）；分 region 与不分 region 变量分开声明；
- Promotion：CPR% 单变量，**非线性**；
- Base 变量：Internal（Coverage：医生数量 / active store / store distribution）+ External（industry trend、competition：social buzz + 竞品活动节点或 competitor sales、seasonality：temperature）。

→ 产品需要一个**模型设计声明（Model Design Spec）** Artifact：模型对象 × 层级 × 变量分组 ×（非）线性形式，作为 3.1/3.2 的输入与 3.4 的对照基准。

## 5. 任务 3.3：模型技术检验（规则完备）

### 5.1 核心原则

> 模型验证不是只看一个指标，而是**多维度交叉验证**。单一指标好不代表模型好，多个指标共同指向同一结论才可信。

| 维度 | 关注问题 | 核心指标 |
|---|---|---|
| 拟合优度 | 模型整体解释力如何？ | R² / Adjusted R² |
| 预测精度 | 预测值与实际值偏差多大？ | MAPE / RMSE |
| 残差独立性 | 残差是否自相关？ | DW (Durbin-Watson) |
| 残差随机性 | 残差是否正态？ | 正态性检验 / QQ 图 |
| 解释力分解 | 各变量贡献是否合理？ | ESS / 方差分解 |

### 5.2 指标判定规则与处理方式

**R²**（Tech Review 模板 benchmark：85%–95%）

| 范围 | 判断 | 建议 |
|---|---|---|
| >0.8 | 高拟合 | 需检查是否过拟合（结合系数合理性，非只看 R²） |
| 0.6–0.8 | 良好 | 可接受 |
| 0.4–0.6 | 中等 | 考虑补充变量 |
| <0.4 | 低拟合 | 重新审视模型结构 |

附加：Adjusted R² 与 R² 差距 >0.1 → 存在冗余变量需精简。

**MAPE**（benchmark：5%–15%）

| 范围 | 判断 |
|---|---|
| <10% | 高精度 |
| 10%–20% | 良好可接受 |
| 20%–30% | 中等，可用于方向判断 |
| >30% | 不可靠，仅用于趋势分析 |

注意：分时段计算（平时 vs 大促）、分季节报告；客户问"准不准"用 MAPE 回答但强调 MMM 用于归因而非预测。

**DW**（benchmark：1.5–2.5，越接近 2 越好）

| 范围 | 判断 | 原因与处理 |
|---|---|---|
| 1.5–2.5 | 无自相关 ✅ | — |
| <1.5 | 正自相关，系数可能被高估 | 遗漏延续性变量（品牌资产/用户习惯）→ 加 adstock/品牌资产变量；遗漏趋势项 → 加时间趋势 |
| >2.5 | 负自相关，系数可能被低估 | 过度差分 → 减少差分；周期模式未捕捉 → 加季节变量 |

**残差正态性**（p > 0.05 为正态）

| 异常 | 原因 | 处理 |
|---|---|---|
| 偏态 | 大促等极端值 | 设大促哑变量 |
| 双峰 | 两个机制（平日 vs 大促） | 分场景/分段建模 |
| 厚尾 | 异常值 | 截尾或稳健回归 |

**ESS / 方差分解**

| 现象 | 判断 | 处理 |
|---|---|---|
| baseline > 80% | 投放解释力弱 | 模型可能有效但增量空间小，建议控制预算 |
| 某渠道贡献 > 50% | 过度依赖单一渠道 | 评估渠道风险，考虑多元化 |
| 分解与业务认知不符 | 模型可能有问题 | 检查变量选择、共线性、变形合理性 |

### 5.3 红旗规则（一票否决，5D 文档 1.5/1.6）

| 红旗 | 含义 | 处置 |
|---|---|---|
| **Negative Base** | 模型认为"零投放时销量为负"，不符合业务常识 | **必须修复后才能使用模型结果**。通常由数据质量、模型设定错误或严重共线性导致 |
| **Negative Coefficient** | 合理预期正向的渠道出现负系数 | 逐一排查：①遗漏混淆变量（竞品压制）②反向因果（销量好才多投）③数据口径问题 |

### 5.4 Tech Review 报告模板（F1）

按模型对象逐一出具（Danone 实例：Online / EC / 社区团购 / O2O 四个对象）：

- 头部：R² / MAPE / DW 实际值 vs Benchmark（85–95% / 5–15% / 1.5–2.5）；
- 明细表：逐月 Actual / Predicted / ERROR / MAPE /（误差差分）²/ ERROR²；
- 年度汇总：Actual vs Predicted vs 年度 MAPE。

### 5.5 Actual vs Predicted 解读范式（标准三维度，5D「技术检验」sheet 范文）

1. **误差 >20% 的关键节点**：定位月份 + 数值 + 原因推测（如年末促销/春节备货未被捕捉、旺季透支效应）；
2. **连续高估/低估的时间段**：识别误差趋势扩大区间，关联到 DW 自相关问题（如对旺季回落节奏判断偏差）；
3. **趋势相反的时间段**：实际与预测背离的区间 + 对应业务场景（短期脉冲式增长、投放衰减效应被高估）。

输出固定以「综合结论 + 核心优化方向」收尾（补充滞后变量/季节因子、校准关键节点权重、补充事件性变量）。

### 5.6 标准话术库（面向客户的解释模板）

规则库为每个指标内置话术，如：

- R²："模型 R² 为 0.75，说明能解释 75% 的销量波动，剩余 25% 主要来自模型未捕捉的外部因素（如竞品突发动作、供应链波动等）。"
- MAPE："MAPE 为 12%，意味着平均预测误差在 12% 左右。考虑到 MMM 主要用于解释归因而非精准预测，这个精度是可接受的。"
- DW："DW 值 1.8，在 2 附近，残差不存在显著自相关，模型没有遗漏重要的延续性变量。"
- ESS："方差分解显示 60% 销量来自品牌积累和自然需求，投放贡献 25%，季节性 8%。增量空间主要在优化投放结构，而非线性追加预算。"

## 6. 任务 3.4：模型业务检验

**定义**（StatusCheckList）：结果是否符合先验。**依赖 3.1 的先验登记表**：

- 各渠道弹性是否落在客户确认的先验区间内（2.31 红黄绿灯规则：绿=在先验范围内；黄=偏离 <30%；红=偏离 >30%）；
- Contribution / ROI Range 是否符合 2.34 预验证形成的预期；
- 与行业 benchmark 对照（渠道 ROI 绿 >2.0 / 黄 1.0–2.0 / 红 <1.0，蚕食率 <30% 健康 / >50% 警惕）。
