## 一、数据智能体是什么（我的理解）

数据智能体是 MMM 流水线的**第二阶段**，夹在「商业理解（S1：因子树已定义）」与「建模（S3+）」之间。

一句话职责：**把散落、异构、脏的原始数据源，变成一张干净、可验收、可建模的统一宽表（Model Input），并在中间设三道质量闸门。**

它解决的核心矛盾是 MMM 项目最痛的环节——数据从几十个口径不一的来源（魔方 / OSS / 德勤手工表 / 尼尔森 / SIA …）汇入，必须对齐到**同一套维度**（模型颗粒度 + 时间颗粒度 + 因子 L1–L8），并且每个指标都要证明「数据可信」且「对模型有用」。

```
       商业理解(S1)                数据智能体(S2)                      建模(S3+)
      因子树/下钻维度  ──▶  2.1 校验 ─▶ 2.2 处理 ─▶ 2.3 交叉校验  ──▶  Model Input
                          (质量闸1)   (ETL集成)   (业务+技术闸2/3)
```

## 二、三道闸门 & 完整漏斗

数据从「很多源文件」漏斗收敛到「一张 Model Input 表」，沿途三次筛选：

| 闸门 | 编号 | 问的问题 | 打分维度 | 处置 |
| --- | --- | --- | --- | --- |
| **闸 1：数据验收** | 2.1 / 2.12 | 原始数据**够不够干净**？ | 一致性·准确性·完整性·颗粒度（4 维，各 0/0.5/1） | =1 验收；=0 不能用并预警；=0.5 人决策 |
| **闸 2：业务交叉校验** | 2.31 / 2.32 | 数据趋势**讲不讲得通业务**？ | 六步业务分析 + ROI/Contribution Range 对照 | 产出「需与客户确认的数据问题」 |
| **闸 3：技术统计检验** | 2.33 / 2.34 | 指标**对模型有没有用**？ | 波动性 CV · 相关性 Pearson · 共线性 VIF | Good≥3 / Acceptable 1.5–3 / 不可用<1.5 |

> ⚠️ 易混点：**2.1 的 4 维打分 ≠ 2.33 的 CV/Pearson/VIF 打分**。
> - 2.1 衡量「**数据本身的质量**」（能不能验收）；
> - 2.33 衡量「**指标对建模的适用性**」（统计上有没有信号）；
> - 2.34 OLS 则在通过前两关的**多个候选指标**里，挑出贡献度/ROI 最贴合业务预期区间的那一个。

**一个 L4 因子常有多个候选指标**（如「品牌传播」下有曝光量、花费）。指标会在 2.33/2.34 被逐步淘汰；**若某 L4 因子仅有的指标全部不达标 → 必须预警**（该因子无可用数据）。

## 三、统一宽表（贯穿始终的数据契约）

整个数据智能体的「通用语言」是这张**长表（long format）宽维度**结构（详见 [`模版化/2.21-宽表数据集维度.md`](模版化/2.21-宽表数据集维度.md)）：

```
模型颗粒度: 品牌 · 品类 · 产品 · 省份组别 · 省份 · 渠道类型 · 渠道
时间颗粒度: 年 · 月
因子映射:   数据类型Level1–Level4 (= 因子树 L1–L4) · Level5–Level8 (= 下钻维度，无则 NA)
度量:       METRICS类型 · METRICS · Unit · VALUE
```

每个数据源经 ETL 后都被「打平」成这套维度的一行行 `(维度…, 指标, 值)`，无法对齐的维度填 `NA`。最终 `Model Data` 表再加上 `Variable no. / Metric no.` 用于建模引用。

## 五、与现有 backend 的对接建议

- `app/agents/validation_rules.py` 目前是 S1 的初步校验；本库的 [`机器可读/validation-scoring.json`](机器可读/validation-scoring.json) 与 [`statistical-scoring.json`](机器可读/statistical-scoring.json) 可作为 S2 校验/筛选的规则源。
- [`机器可读/wide-table-schema.json`](机器可读/wide-table-schema.json) 与 `app/mmm/pivot` 的输出契约对齐，可作为「数据集/Model Data」列定义的单一事实源。
- [`机器可读/factor-ranges.json`](机器可读/factor-ranges.json) 把因子树的 ROI/Contribution 预期区间机器化，正是 2.34 OLS 筛选与建模先验（prior）的输入。
- [`机器可读/workflow.json`](机器可读/workflow.json) 描述 2.1→2.35 的 DAG，可校验 blueprint 中数据智能体任务的完整性。

> 数值与区间均来自真实 Danone Mizone 案例，仅作为模版示例；新项目应替换因子树、ROI 区间与数据源清单。
