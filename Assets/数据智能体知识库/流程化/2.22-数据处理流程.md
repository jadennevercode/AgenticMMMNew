# 2.22 数据处理流程（流程化 / 人交互确认数据处理逻辑）

> 类别：Input ｜ 数据处理阶段的执行流程、检查步骤与 TaskLog 管理。出处：`Data Process_2.22`。

## 数据处理 7 步流程

| 步 | 动作 | 说明 |
| --- | --- | --- |
| 1 | 初步梳理数据逻辑 | 按数据逻辑总表（TaskLog）确认每个 task |
| 2 | 定义澄清 & 补充数据 | 与业务/数据对接人澄清口径、补缺 |
| 3 | 数据补充后再梳理逻辑 | 迭代修正 processing logic |
| 4 | by task 导出数据后检查 | 4-a 逻辑/字段分类/名称是否匹配（**一天 4–5 张表**）；4-b 初步检查趋势是否缺失；4-c 判断是源数据问题还是 mapping 问题 |
| 5 | 逻辑更新后重新导出并 charting | 为交叉校验做准备 |
| 6 | 与源数据 cross-check 一致性 | **需两人互相检查** |
| 7 | 寻找异常点原因 | 定位异常 |

## 数据检查 5 步（4-a 细化）

1. 据 TaskLog 确认 task 编码/名称/优先级，按优先级定检查顺序。
2. 检查对象 = 共享文档 `all_dataset` 中对应 task。
3. 校验对象 = 据 TaskLog 数据源确认（在 manual-德勤 还是 OSS）。
4. 检查内容①：all dataset 中 charting 的部分（**要用真实数据源检查，不能直接用 charting 表的数**）。
5. 检查内容②：每个 task 的年度大数、增长率、月度趋势及其他准确性检查。

## TaskLog 字段（任务台账）

`优先级 | NO | TASKNAME | 数据来源 | 数据逻辑对接人 | 数据问题 | 数据反馈 | 数据处理 | LOG | PROCESSING LOGIC(按日期迭代) | Source_Table | Task_Summary | 处理进度 | Owner | 备注`

- 优先级分 1–4 级，决定处理与排期顺序（见 Timeline 甘特）。
- 数据来源二分：`Manual-德勤`（手工表）/ `OSS`（数据库）。
- 本案共 **37 个 task**，覆盖 EC / O2O / ANP / Media / Social / Trade / SIA / Macro 等。

## KBQ 因子 ↔ Task 对应

每个 task 回链到因子树（L1–L4）+ 指标选型 + 内部讨论备注 + 颗粒度/完整性/一致性/准确性标注。
示例：`Task34&35` ↔ `Baseline Factor / External Factors / Industry Trend / NAB品类整体趋势` ↔ 指标 `NAB All Channel Offtake Volume`。

> 备注里常见三类标签：`[需沟通]`（待与客户/数据方确认）、`[模型支持]`（需共线性/显著性检验，承接 2.33）、`[唐老板建议]`（专家业务判断，承接 2.31）。
